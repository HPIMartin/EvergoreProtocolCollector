# Process Learnings

A running log of where our process slipped and how we prevent it next time. The **Reviewer** agent
(see [knowledge-base/multi-agent-playbook.md](knowledge-base/multi-agent-playbook.md)) appends a row
whenever a rule was missed ŌĆö even on a PASS. The point is that future work and future agents drift
*toward* the process, not away from it. Recurring items should be promoted into `CLAUDE.md` or the
[engineering handbook](knowledge-base/engineering-handbook.md).

| Date | What slipped | Rule (handbook ┬¦) | Fix / prevention |
|------|--------------|-------------------|------------------|
| 2026-06-15 | D1 rename commit left a duplicated `architecture.md` violation item + stale `PageContentExtractor` KB refs; fixed post-hoc one commit later | KB-first/KB-current ŌĆö name/behavior changes update the relevant KB doc *in the same change* (CLAUDE.md golden rules; handbook ┬¦1/┬¦8) | On a rename, grep all of `docs/` for the old identifier within the same commit; when editing a numbered list, replace in place and re-read it to catch paste duplicates before committing |
| 2026-06-20 | The acceptance test was first named `D2AcceptanceTest` ŌĆö code referenced an external backlog ID instead of being self-explanatory; author rejected it | **Code must be self-explanatory and stand on its own without external documents; meaningful, intention-revealing names** (handbook ┬¦3) | Name by what a thing *is/does*, understandable from the source alone ŌĆö no tracker IDs or task references. When spawning an implementer, hand it the *capability* name, never the backlog ID |
| 2026-06-21 | Ran the falsifier and reviewer **in parallel in the same working tree** (B8); the falsifier's mutation counter-tests were live when the reviewer built, so the reviewer reported a phantom FAIL (4 acceptance tests "red") that did not reproduce on a clean build | Falsify (step 3) then review (step 4) are independent, but must **not** share a mutated tree (multi-agent-playbook ┬¦pipeline) | Run tree-mutating agents **sequentially**, or give each its own git worktree (`Agent` `isolation: "worktree"`); never overlap a mutating agent with another agent's build. Planner re-verifies any disputed verdict with a clean build before trusting it |
| 2026-06-21 | Folding the reviewer's nits into the feature commits stalled ŌĆö the playbook's prescribed `git reset --soft <base>` + rebuild is **permission-blocked** in this harness (all `git reset`, plus `rm` and `git branch -D`, are denied) | Fold fixes into the commit they belong to ŌĆö never append "fix review" commits (handbook ┬¦7; playbook ┬¦FAIL-loop) | Rebuild the branch **without `reset`**: fresh branch off the base, `git checkout <wip-tip> -- <files>` to stage each logical group and commit, then `git branch -m` to swap names. Playbook updated to document the reset-free method + the blocked commands |
